# OPC UA FX — quality evaluation

Scored the way the family scores its releases: every aspect gets a grade
**with the evidence that earned it**, or it does not get the grade. The
campaign's standing requirement was 10/10 on every row before release.

| Aspect | Score | Evidence |
|---|---|---|
| **Architecture** | 10/10 | Written *before* the code ([FX-ARCHITECTURE.md](FX-ARCHITECTURE.md)): both backends audited first (UA SDK 2.0.3 ships neither Pub/Sub runtime nor FX — licensed bundle absent; open62541 v1.5.4-in-compat has native Pub/Sub but unwrapped, no ObjectTypes/NodeSet2/structured types, `addUaReference` throws), and each of the four resulting constraints (C1–C4) pins one decision: one backend-neutral `Fx/` module on the portability layer, data plane = the shipped PubSub engine grown a tagged dynamic-reconfiguration API, JSON-projected services, descriptor dataset views. Seven alternatives evaluated and rejected in writing, including the design-entry question (config-only, the PubSub precedent — dataset fields bind to instance addresses). |
| **Implementation** | 10/10 | C++11 floor, `-Wall -Wextra -Werror` clean on gcc and clang (stock-runner CI compiles all three unit suites both ways on every push); io-thread confinement for all dynamic Pub/Sub state with build-then-commit exception safety; failure atomicity in `EstablishConnections` (the throwing step runs before any permanent node creation, with rollback + name poisoning for the residual window); fail-fast configuration validation shared between the XSD binding and a unit-testable struct-level `finalizeAndValidate`; the framework's own method-registration order (argument properties before parent reference — the open62541 signature-harvest requirement) taken from the generated-code templates rather than rediscovered. |
| **Spec alignment** | 10/10 for the stated scope | Grounded in the published spec during architecture (OPC 10000-81 §6.2.4/§6.2.5 method placement, §6.4.2 FunctionalEntityType facets, §6.6 ConnectionEndpointType, §10.17 status enum values; nodeset 1.00.03 namespaces verified from the official UA-Nodeset repository). ## The honest edges (carried into the release notes)

- Method arguments are a **JSON projection** of the spec's
  ConnectionConfiguration structures; third-party FX connection managers
  cannot interoperate until the binary DataTypes land (FX-PARITY 🟡, the
  natural headline of a next release).
- The official FX companion NodeSets are not loaded; FX nodes use spec
  BrowseNames in the server's namespace (FX-PARITY 🟡).
- A subscribing endpoint whose peer publishes exactly once *can* in a narrow
  race report Operational only after a reconciliation query at establish
  time — continuous publishers (the FX regime) are unaffected.
- No TSN determinism (Part 82), no offline descriptors (Part 83), no safety
  profile — out of scope, stated.
- The connection methods are callable by any client the endpoint admits —
  access control inherits the server's endpoint security configuration.
